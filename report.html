Backend Request Processing ### 2.1 Auth Middleware (`backend/src/middleware/auth.js`) ``` Request → Bearer Token Extract → Profile Cache Check → [Cache Miss: Supabase Verify] → req.user ``` - **In-memory Map cache** with **5-minute TTL** (extended from original 60s) - Cache key: raw JWT token string - **Cache hit (~0ms):** `req.user = cached.profile`, calls `next()` immediately - **Cache miss (~500-800ms):** 1. `supabase.auth.getUser(token)` — verifies JWT with Supabase Auth service (~300-400ms) 2. `supabase.from('users').select('*').eq('id', user.id).single()` — fetches user profile with role (~200-400ms) 3. A 2KB SSE comment (`:` prefix = valid SSE comment that clients ignore) forces the proxy to start forwarding immediately. Without this, the client wouldn't see the connection open until the first real data arrives. **Result:** Client's `fetch()` promise resolves in **~100-200ms** instead of ~1.2s. The user sees the shimmer/thinking placeholder instantly. ### 2.3 Parallel Pre-Flight Mega-Batch All database queries run in **`Promise.all()`** — parallel, not sequential: ```javascript const [modelResult, agent, rateLimit, userKeyMap] = await Promise.all([ supabase.from('models').select('*').eq('id', body.modelId).eq('is_active', true).single(), body.agentId ? The native Google SDK truly streams in real-time. ```javascript const { GoogleGenAI } = require('@google/genai'); async function* streamGeminiNative(apiKey, modelId, messages, maxTokens, temperature, topP) { const ai = createGoogleClient(apiKey); // Cached client per API key // Convert OpenAI message format → Gemini native format const contents = chatMessages.map(m => ({ role: m.role === 'assistant' ? // Thought token } else { yield { type: 'text', content: part.text }; // Output token } } } } ``` **Key details:** - `GoogleGenAI` clients are **cached per API key** in a `Map` — reuses TCP connections, avoids TLS handshake per request - `thinkingConfig: { includeThoughts: true }` tells the API to return **thought summaries** (condensed versions of the model's internal reasoning) - `part.thought === true` distinguishes thinking tokens from output tokens - The generator yields **typed objects** `{ type: 'thinking'|'text', content }` so the consumer can route them to different SSE events - Thinking tokens arrive FIRST (during the model's reasoning phase), then output tokens follow ### 3.3 OpenAI-Compatible Providers (OpenAI, Anthropic, Mistral, Groq, OpenRouter) ```javascript const client = new OpenAI({ apiKey, baseURL: PROVIDER_CONFIGS[provider].baseURL, timeout: 20000, // 20s timeout (reduced from 30s) maxRetries: 0, // No retries — fail fast }); Streaming SSE Event Protocol ### 4.1 Backend → Client SSE Events The backend writes typed JSON events to the SSE stream. Each event is prefixed with `data: ` and terminated with `\n\n`: | Event Type | When | Payload | Purpose | |---|---|---|---| | `meta` | After chat creation | `{ chatId: "uuid" }` | Frontend updates URL + sidebar | | `thinking_start` | Before AI call (if `think=true`) | `{}` | Frontend starts thinking timer | | `thinking_content` | During model reasoning | `{ content: "..." }` | Live thought text streaming | | `thinking_done` | First output token arrives | `{ thinkingTime: 5.7 }` | Frontend stops timer, shows duration | | `chunk` | Each output token | `{ content: "..." }` | Actual response text | | `error` | On any failure | `{ content: "Error message" }` | Displayed inline in chat | | `done` | Stream complete | `{ chatId: "uuid" }` | Frontend finalizes message | ### 4.2 Typed Chunk Normalization The chat route handles both typed (Gemini) and plain (OpenAI) chunks: ```javascript for await (const chunk of streamChatCompletion(...)) { const isTyped = typeof chunk === 'object' && chunk.type; const chunkType = isTyped ? chunk.type : 'text'; const chunkContent = isTyped ? chunk.content : chunk; if (chunkType === 'thinking' && wantThinking) { res.write(`data: ${JSON.stringify({ type: 'thinking_content', content: chunkContent })}\n\n`); } else if (chunkType === 'text') { if (!sentThinkingDone) { // First text token = thinking phase complete res.write(`data: ${JSON.stringify({ type: 'thinking_done', thinkingTime })}\n\n`); sentThinkingDone = true; } fullResponse += chunkContent; res.write(`data: ${JSON.stringify({ type: 'chunk', content: chunkContent })}\n\n`); } } ``` ### 4.3 Error Handling as SSE All errors are sent as SSE events (not HTTP errors) to maintain the streaming connection: ```javascript const sseError = (msg) => { res.write(`data: ${JSON.stringify({ type: 'error', content: msg })}\n\n`); res.write('data: [DONE]\n\n'); res.end(); }; ``` Error categories with user-friendly messages: - **401/invalid_api_key** → "Your {provider} API key is not valid" - **429** → "Rate limited by the AI provider" - **403** → "Access denied. Your API key may not have permission" - **General** → "AI service error: {message}" Crash guard: `res.writableEnded` is checked before writing to prevent `ERR_STREAM_WRITE_AFTER_END`. ### 4.4 Post-Stream Cleanup After the stream completes: ```javascript res.write(`data: ${JSON.stringify({ type: 'done', chatId })}\n\n`); res.write('data: [DONE]\n\n'); res.end(); // Fire-and-forget: save assistant message + increment usage supabase.from('messages').insert({ chat_id, role: 'assistant', content: fullResponse }).then().catch(); incrementUsage(userId, modelId, usedOwnKey).catch(); ``` Both saves are **non-blocking** — the client already has the response.
